#+TITLE: Local State in Event-centric Games
#+AUTHOR: Pancake
#+DATE: 2026-07-06
#+DESCRIPTION: After building rules of composition, we turn out attention to another problem: Representing state.

In the [[file:Games-as-Graphs.org][previous post]], we described games as graphs of event
transformations. This shifted the focus from ordered execution toward
compositional behaviour, where interactions are defined by how events
produce other events.

This representation captures causality, but leaves state outside the
structure. State is still required for transformations, yet it is not
organised by the graph. Instead, it is implicitly shared across the
system, accessed without regard to causal boundaries.

This post introduces a way to localise state boundaries within the
event structure while preserving a single, global view of the
world. The result is a model where state becomes part of the same
compositional system as events, rather than an external layer. More
importantly, treating persistent state as part of the structure of the
game lets useful invariants to emerge automatically. While they are
only a byproduct of the representation introduced here, these
invariants become the key ingredient in constructing a safe
asynchronous scheduler in the next post.

** Localised Access to a Global World: Projections

In games, most interactions act on natural data boundaries. For
example, a physics step doesn't need to know about UI rendering
details. Likewise, the UI state doesn't depend on the states of
physics bodies. Each system acts on a narrow context, even if the
system as a whole contains more information. This suggest that
distinct pieces of data live alongside one another, which can be
collected in one tuple representing the state of the whole game.

To illustrate, let's reintroduce the game demo from the previous post:

#+INCLUDE: ".game-graph-demo.html" export html

This game has a collective state of:

~(Balls, Surfaces, PhysicsSlot, SurfaceUI)~.

Yet, each system acts on various sub-tuples of the whole world.
 - Placing balls only acts on ~Balls~
 - Placing surfaces acts on ~Surfaces~ and ~SurfaceUI~
 - Creating a physics task acts on ~Balls~ and ~Surfaces~, but
   registering, executing, and polling the task only requires
   ~PhysicsSlot~.

To bridge the gap between local access and the global source of truth,
a /mask/ can be constructed. For example, if placing a surface only
requires ~Surface~ and ~SurfaceUI~, a mask for this access can be
constructed as:

~(Skip, Keep, Skip, Keep)~

The mask represents an access pattern to the global state
tuple. However, the mask yields no information on its own, it only
specifies which components are visible. Applying the mask requires a
/projection function/, which extracts the slots of data marked ~Keep~,
leaving out the other components. The workflow is illustrated below:

Importantly, masks are defined solely from the global world
state. Once that state has been fixed, every projection is determined
independently of the event graph itself. Consequently, introducing
projections does not change the existing rules of event composition;
it merely restricts which portion of the global state each
transformation may observe.

** Masks in Practice

So far, it seems like masks are more fundamental than pieces of
state. Instead of accessing ~Balls~, interactions now access
~(Keep, Skip, Skip, Skip)~. However, maintaining data dependencies
in code through magic tuples is not helpful. In a complex game with
dozens of distinct components of data, a single tuple can not only
be cumbersome to type, but it also becomes error prone. Since the
tuple specifies nothing but /indices/ to a global state, game iterations
can misalign the indices, causing access to the wrong pieces of data.

To address this issue, we need a systematic way to construct the mask
tuples from local data automatically. Rather than specifying a magic
tuple, we may specify ~(Balls, PhysicsUnit)~, and get back,
~(Keep, Skip, Keep, Skip)~ automatically.

Most static languages model this by mapping data to IDs or enums, then
defining data requirements by referring to these enums. Some languages
offer /reflection/, which allows direct inspection of types, avoiding
the manual mapping to IDs.

In /Idris/, the language used to implement the game demo, there is a
third option which completely hides the construction of masks from the
end user. Idris can automatically construct data whose structure
follows deterministic rules. Since a mask is completely determined by
the local requirements and the global state tuple, Idris can construct
it automatically once the construction rules are provided.

To introduce the construction rules, the implementation defines a
~SubList small great~, which states that ~small~ is a valid local state for ~great~.
If this statement holds, then Idris constructs an instance for ~SubList~
automatically, and the instance is a mask on ~great~ that isolates
components requires for ~small~. The instance isn't constructed
arbitrarily, the rules must be presented concretely:
 1. While ~small~ has components, take the first component.
 2. If the first components of ~small~ and ~great~ match, pop both and
    emit ~Keep~
 3. If the first components don't match, pop only ~great~ and emit
    ~Skip~.
 4. Once ~small~ is empty, emit ~Empty~, which skips the remainder of
    ~great~. If ~great~ runs out first, then fail construction, since
    ~small~ is then not a valid local state for ~great~.

Below is a snippet from the Idris REPL that demonstrates the
construction:

#+BEGIN_SRC idris
λΠ> let mask : SubList [List Surface, SurfSpawner] [List Ball, List Surface, PhysicsSlot, SurfSpawner];
        mask = %search; -- Ask Idris to search an instance
    in mask
Skip (Keep (Skip (Keep Empty)))
-- ^- Linked list: Skip -> Keep -> Skip -> Keep -> Empty
#+END_SRC

Thus, when using the Idris implementation, specifying the mask isn't
necessary. Idris infers the types of ~small~ and ~great~ from context,
then construct ~SubList~ automatically, hiding the masks and
projections as an implementation detail. Furthermore, since the masks
are constructed at compile-time, they remain type-safe and their
construction incurs no runtime cost.

** Persistent Structure

In the current system, events are transient. They arise, trigger
reactions, then disappear. State, on the other hand, is
persistent. Inevitably, any property that remains true throughout
execution is expressed as state.

Thus, the projection system also has an unexpected side consequence.
By enforcing that all state is a projection from a global state tuple,
we also unintentionally enforced that all persistent properties of the
game live in a tuple.

This makes the tuple of the world state more than an implementation
detail. The tuple also describes structural properties that cannot
change during execution. For example, if the game defines that the map
is a ~Vect 10 Surface~, then it is structurally fixed that the game
carries only 10 surfaces. Interactions can modify the contents of this
vector, they can ignore some or all surfaces, but they can't construct
~15~ surfaces, as ~Vect 10~ doesn't have enough space for it.

In the [[file:Game-Loops.org][next post]], this system is used to express structured task
scheduling, where a statically fixed budget will be allocated to
queues of various tasks. This allows the performance of a game to be
predictable. For example, fixing ~PhysicsSlot~ as a job slot of ~512~
budget like ~JobSlot 512 PhysicsTask~, the game cannot execute more
than 512 physics interactions per-frame.

** Conclusion

In this post, we extended the event graph with a notion of localised
state. Rather than treating state as a globally accessible resource,
each interaction now operates on a projection of a single world
state. Masks describe these projections independently of the event
graph, allowing state locality to be introduced without changing the
compositional structure established in the previous post.

More importantly, this representation reveals that persistent state
carries structural information about the game itself. Since events are
transient while state persists, any invariant that should hold
throughout execution naturally becomes part of the world
representation. These guarantees are not introduced through additional
runtime mechanisms, but emerge from the organisation of state itself.

However, one significant gap remains. Although interactions now
compose over both events and localised state, our composition model
still describes only acyclic propagation of behaviour. Many gameplay
systems, including asynchronous job execution, deferred work, and
iterative processing, require behaviour that feeds back into the
system over time. In the [[file:Game-Loops.org][final post]] of this series, we introduce the
primitives required to express these cyclic interactions, showing how
event graphs can model bounded asynchronous execution while preserving
the structural guarantees developed here.
